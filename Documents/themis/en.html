Before start

Themis is not a package manager, it uses local package managers on the basis of themis purpose Instead of packaging simple builds separately for multiple platforms, you just need to unpack their dependencies. It's a multi-purpose cross-script management tool that you just need to specify as variables (NOT first release There is no version management system for.

Requirements:

bash, python3/pip3(opt), ruby/gem(opt), wget, curl, gzip/tar

Installation:

Arguments and their intended usage
  • --help, help, --yardım, yardım, -h

    • The help argument prints the arguments from the other tool and their intended use.

  • --makepackage, makepackage, --paketyap, paketyap, -mp

    • with the package argument, Packages the projects prepared according to Package preparation procedure. Verifies the directory(s) you specify as parameter(s) first, if it conforms to the procedure, it compresses them with tar.gz(gzip) and saves the command to the directory where you run it. The important thing is that the saved package does not have a "_" character in its name and the filename is contiguous.

  • --localinstall, localinstall, --lokalyükle, lokalyükle, -li

    • You can install themis package files that you compressed with the load locally argument, this argument must be run with root privileges, but themis does not have a package compilation system of its own, it detects the functions of the bash scripts and the procedure and installs your package(s) accordingly.

  • --install, install, --yükle, yükle, -i

    • The install argument should be used with root rights. It basically uses the functions of the local install. It works with catalog logic. The indexes of your repositories are located in the/usr/share/themis/index.sh file, you can add and remove the repositories you want you can access it from Create, add, and remove a repository.

  • --uninstall, uninstall, --kaldır, kaldır, -u

    • With the remove argument, you can remove (delete) the files and directories installed on your system with themis, but the producers are not responsible for the damage to your system due to the gaps in the metadata files of the packages. Please stick to the package construction procedure and respect other people, always stay with free software.

  • --updateindex, updateindex, --indeksgüncelle, indexgüncelle, -up

    • themis works with catalog logic, it pulls information from the settings files saved in the file system beforehand, so you can access the information of the packages in the index you bought both fast and even when you are offline, the main index file where the repositories are hosted is specified as arrays of the repositories variable in the /usr/share/themis/index.sh file, It saves to dosyasının içerisinde repositories değişkeninin arrayları olarak belirtilir depolardan çekilen doğrulanmış paket bilgileri ise /usr/share/themis/repositories dizinine kaydolur. It saves to /usr/share/themis/repositories. It is enough to run this argument with sudo, there are no parameters or additional arguments at the moment, repository assignments can be made manually or with scripts that you have prepared or will prepare.

  • --list, list, -l

    • When the list argument is used plainly, it lists the names of the packages installed on your system. If you use --repositories, repositories, repository, repo as the second argument, you can see the package information in the repositories whose index you have updated before.

  • --info, info, --bilgi, bilgi, -i

    • You can see the information of the package(s) with the information argument, in order: package, version, developer, description LICENSE and README files, if any

  • --version, version, -v

    • You can see the version of themis we have used with the version argument and you can reach us from the link addresses..

Package preparation procedure

How is the package prepared?

First of all, open a metadata file named CONTROL in the main directory of your project and enter the package, version, maintainer, description variables in this file as mandatory information The package variable is absolutely required and the package variable cannot contain characters such as "_" and " ", then you have to assign the dependencies, the dependencies are assigned as systembase_depends="" according to your operating system base You can use existing dependencies such as debian_depends, arch_depends, fedora_depends, pisi_depends, opensuse_depends. You can also assign dependencies for python and ruby ​​modules using python3_depends, ruby_depends variables Along with these, there are postinst scripts that come after the dependencies. After the dependencies are installed, you can specify a single script that will work according to the base of the system or on all systems, but do not forget that postinst scripts are not reversible, only during the installation phase. they perform the same functions as dependencies and they have to be deleted from the system manually. Do not use postinst scripts unless necessary. -inst.sh, opensuse-inst.sh, post-inst.sh Now let's come to the most important build() { ... } function to be installed in this function, but with other sub-functions previously assigned to the build function, you can download files from the internet with the help of curl or wget in the build with gcc/cc You can compile files in subdirectories, but the main point here is When moving any directory, you should use the move "./source" "/target" function instead of the mv command, so that the files installed on the system will be deleted in the reverse engineering process, while creating any directory or directories is based." dir" "dir" important point please do not use directories such as /, /bin, /usr, /dev, /opt, /boot, /etc is that you don't define it directly because directly defined directories are deleted in reverse engineering process and if it is system directory this can cause major damage to your system.

Create, add, and remove a repository

Create a repository

You can create the repository on any remote server. Open a file named inde.sh in the main directory where the files are located and create maintainer, reponame, packages variables in it. reponame is required as arrays inside packages as follows make an assignment file.tar.gz_package_version However, remember that the first array, namely packages[0], should be the link where the index is located and the file.tar.gz files are handled as root. To explain this with an example, the only thing that should not be forgotten here is that the "_" character will be used as a separator, so "_" should not be in the names of the files.

Adding and removing a repository

To add a repository, copy the link containing the index.sh file of your repository without removing the index.sh and put repositories in the /usr/share/themis/index.sh file in your file system. add it as a new array to the variable, in the same way in the removal process, remove the array with the link from the file /usr/share/themis/index.sh and /usr/share/themis/repositories Go to the directory and delete the metadata files of the relevant repository, don't forget to update the index after this process is finished
